## 1. WHAT THIS PLATFORM IS

Otima Energia is a technology-driven energy brokerage platform for the **Brazilian free electricity market (ACL)**. It combines:

- A **marketing website** (Portuguese, public-facing)
- A **CRM-lite / Deal OS** system for managing client leads, document intake, bill parsing, supplier quotes, and commissions
- A **Client Portal** for self-service document upload
- An **ECOS engine** for AI-driven energy contract analysis

The platform follows a **Bill-First Architecture**: energy bills (PDFs) are the foundational input. Everything else — client profiles, ECOS analysis, dossiers, RFQs — flows from successfully parsed bills.

## 4. DEAL IDs — HOW THEY WORK

- **Format:** UUIDs generated by PostgreSQL (`gen_random_uuid()`)
- **Example:** `e6ca5d58-53c8-4a03-93ba-7cde00b65805`
- **Assignment:** Automatic on database insert — the application never generates the ID manually
- **Two creation paths:**
  1. **Zoho Intake** (automated): `POST /api/intake/zoho/deal` — Zoho CRM posts lead data, portal creates the deal
  2. **Manual** (admin): `POST /api/deals` or `POST /api/clients/:clientId/deals` via Admin UI

## 5. THE DEAL JOURNEY (State Machine)

Every deal follows this progression:

```
DRAFT → RFQ_SENT → QUOTES_RECEIVED → OFFER_SELECTED → ONBOARDING_PENDING → CONTRACT_SENT → CONTRACT_SIGNED → SUPPLY_LIVE → CONTRACT_ENDED → CLOSED
```

**Special states:** `LOST` (accessible from almost any state)

Each transition:
- Is validated against `DEAL_STATE_TRANSITIONS` map
- Is logged in `deal_state_transitions` table with actor, reason, and timestamp
- Automatically sets lifecycle timestamps (e.g., `contractSignedAt`)

### Assembly Line (Visual Pipeline)
The Assembly tab shows the deal's progress through these stages:
1. **Origin & Qualification** — Lead enters system
2. **Bill Uploaded** — Energy bill PDF uploaded and parsed
3. **ECOS Generated** — Market analysis snapshot created
4. **Client Dossier** — Energy profile assembled (DRAFT → READY → LOCKED)
5. **Dossier Locked** — Data frozen for RFQ
6. **RFQ Sent** — Quotes requested from suppliers
7. **Quotes Received** — Supplier responses collected

## 6. THE BILL-FIRST PIPELINE

### How Bill Upload Works
1. User uploads PDF via Assembly tab → `POST /api/deals/:dealId/bills/upload`
2. File stored in GCS with key prefix `bills/`
3. A `BILL_PARSE` job is enqueued in the job runner
4. Job runner downloads the file, calls the VPS parser
5. Parser returns extracted fields (CNPJ, UC code, consumption, demand, tariff info)
6. `normalizeParserResponse()` maps raw parser output to canonical schema
7. Results saved to `bills_extracted` table

### Client-Side Upload Retry
- **3 attempts** with exponential backoff (1s, 2s, 4s delays)
- **Only retries** on network errors and 5xx responses
- **4xx errors fail immediately** with categorized messages:
  - 404 → "Endpoint not found" (parser URL issue)
  - 413 → "File too large"
  - Timeout → "Parser timeout"
  - Network → "Network error"

### Parser Service
- **URL:** `https://parser.otimaenergia.com`
- **Auth:** `X-Parser-Key` header (from `PARSER_API_KEY` env var)
- **Timeout:** 300,000ms (5 minutes)
- **Health endpoint:** `/health` (checked by admin diagnostics panel)
- **Handles:** Bills (STANDARD_BILL, CONSUMPTION_HISTORY, NF3E, DEMONSTRATIVO) and PRCs

### Key Parser Data Fields
The normalizer maps dozens of field variations to canonical names:
- `customerId` (CNPJ) — with redaction handling ("N/A (redacted)" → REDACTED badge)
- `ucCode` (Unidade Consumidora)
- `grupo` / `subgrupo` (tariff classification)
- `consumoPontaKwh` / `consumoForaPontaKwh`
- `demandaContratadaKw` / `demandaMedidaKw`
- `bandeira` (tariff flag — mapped from `tariffFlag`/`bandeiraName`/`bandeiraColor`)
- `totalAmount` / `totalEnergyKwh`

**Brazilian Number Parsing:** `parseBrazilianNumber()` handles dots-as-thousands (e.g., `96.199,73` → 96199.73, `1.234` → 1234)

## 7. ECOS SYSTEM (Energy Contract Operating System)

### What It Does
Evaluates a client's energy costs against market benchmarks to determine if they're overpaying.

### How Snapshots Are Generated
1. Requires at least one parsed bill
2. Captures frozen client data (UC, submarket, consumption, distributor)
3. Matches against PRC pricing data in `canonical_pricing_rows` / `market_price_benchmarks`
4. Calculates client price vs. market average ratio
5. Assigns a 5-band market position

### 5-Band Market Position
| Band | Ratio (Client/Market) | Meaning |
|------|----------------------|---------|
| WELL_BELOW | ≤ 0.90 | Great position, no urgency |
| BELOW | 0.91 - 0.97 | Competitive, monitor for next cycle |
| ON_PAR | 0.98 - 1.03 | Market average |
| ABOVE | 1.04 - 1.10 | Overpaying, run RFQ recommended |
| WELL_ABOVE | > 1.10 | Significant savings potential, urgent |

### ECOS PDF
- Generated via HTML template → browser `window.print()` (not Puppeteer)
- Preview: `GET /api/deals/:dealId/ecos/:snapshotId/preview`
- Shows: gradient header, executive summary, visual band chart with "Voce" marker, consumption stats, CTA
- Client-facing (no raw PRC data exposed)

### PRC (Price Reference Circulars)
- Uploaded via PRC Upload Center or PRC Management page
- Parsed by VPS parser → rows stored in `prc_rows`
- Canonical submarkets: `SE_CO`, `S`, `NE`, `N`
- Canonical products: `CONVENCIONAL`, `INC_I50`, `INC_I100`, `INC_I0`
- PRC rows can be inline-edited in PrcManagement (edit button per row)
- Statuses: `PARSED` (≥12 rows), `NEEDS_REVIEW` (1-11 rows), `FAILED` (0 rows)

## 8. ZOHO CRM INTEGRATION

### How Leads Enter the System
1. Zoho CRM sends lead data to `POST /api/intake/zoho/deal`
2. Authenticated via `x-zoho-intake-key` header
3. System checks for duplicate leads (by `zohoLeadId`) — idempotent
4. Matches or creates a client record (by CNPJ or company name + email)
5. Creates a deal in `DRAFT` state

### Auto-Assignment Rules
- Group "B" (low voltage) → assigned to **Callum**
- "Hotkey" outcome → assigned to **Renan**
- Default → **Callum**

### Field Mapping (Zoho → Portal)
| Zoho Field | Portal Field |
|-----------|-------------|
| `zohoLeadId` | `zohoLeadId` |
| `companyName` | `companyName` |
| `email` | `email` |
| `cnpj` | `cnpj` |
| `brMarket` | `brMarket` (ACL/ACR/Unknown) |
| `brGroup` | `brGroup` (A/B/Unknown) |
| `outcome` | `zohoLeadOutcome` (Hotkey/Warm/Tepid) |
| `sourceAgent` | `zohoLeadSourceAgent` (Clara/Sophia/Unknown) |
| `dmName` | `dmName` (Decision Maker) |

### Background Jobs After Intake
- `ZOHO_SYNC_SNAPSHOT`: Syncs tasks, calls, notes from Zoho
- `ZOHO_CREATE_CALLBACK_TASK`: Creates a callback task in Zoho CRM, scheduled within business hours

### Where Zoho Data Appears in UI
- **Overview tab** (blue banner + Lead Summary card) — shows lead ID, outcome, agent, market, decision maker info
- **Sales tab** — read-only mirror of Zoho CRM activity

### Error Handling
- Failed intake attempts stored in `zoho_intake_errors` table
- Admin UI: ZohoIntakeErrors component for monitoring

## 9. CLIENT PORTAL & INTAKE SYSTEM

### How It Works
1. Admin clicks "Generate Link" in Deal Tracks tab
2. System creates an `upload_session` with a 32-char secure token and optional 6-char access code
3. Email sent via Zoho SMTP to client with portal URL + access code
4. Client accesses portal, enters access code, goes through guided workflow

### Portal Workflow Steps
1. **Access verification** — token + access code validation
2. **LGPD consent** — digital privacy agreement (stored in `consent_records` with IP/timestamp)
3. **LOA signing** — Letter of Authorization (checkbox or drawn signature), PDF generated via `pdf-lib`
4. **Bill upload** — multi-file upload for energy bills (default target: 6 bills)
5. **Completion** — session marked as used, chase stopped, internal notification triggered

### Document Chase System
- **Manual trigger** for intake link delivery
- Chase state tracked in `track_chase_state` table
- Automated reminders via external webhook (`CHASE_WEBHOOK_URL`)
- Auto-stops when client completes intake

## 10. WHATSAPP INTEGRATION (Current State)

**There is NO direct WhatsApp API integration** (no Twilio, no Meta Business API). The current approach is:

### What's Built
1. **Click-to-chat links** (`wa.me/...`) — operators click a button to open WhatsApp Web with pre-filled messages
2. **Outbound webhooks** — the system fires webhooks to an external automation service (Make.com, n8n, or similar) for:
   - `chase/start` — trigger automated reminders
   - `chase/stop` — stop reminders when intake is complete
   - `chase/update` — update chase state
3. **WhatsApp opt-in tracking** — `whatsapp_opt_in` and `contact_whatsapp` fields on contacts (validated for E.164 +55 format)
4. **Message templates** — suppliers have configurable WhatsApp message templates for RFQ distribution

### What's NOT Built
- No server-side automated WhatsApp message sending
- No Twilio/Meta API integration
- The webhook listener (external service) needs to be separately configured to actually send WhatsApp messages

## 11. RFQ & COMMISSION SYSTEM

### RFQ Flow
1. **Dossier** must be marked `READY` (required fields: legal name, CNPJ, distributor, annual consumption, eligibility)
2. Dossier gets `LOCKED` when RFQ is created (snapshot preserved)
3. **Blind Auction Panel** — select suppliers, send RFQs via multiple channels (email, WhatsApp, portal, phone)
4. **SLA tracking** — response time monitoring, overdue marking
5. **Quote recording** — quotes linked to specific RFQ dispatches

### Commission OS
- **Milestone-based tracking** (50/50 payment model):
  - **M1** (50%): Triggered by `CONTRACT_SIGNED`
  - **M2** (50%): Triggered by `SUPPLY_LIVE`
- **Statuses:** `FUTURE` → `PENDING` → `INVOICED` → `PAID` (also `OVERDUE`, `DISPUTED`)
- **Reconciliation:** Compares estimated vs. actual values, creates `ADJUSTMENT` events
- **Notifications:** Automatic alerts for overdue commissions

## 13. JOB RUNNER

A lightweight polling-based background job system:

- **Poll interval:** 30 seconds
- **Idempotency:** `enqueueJobIfNotExists` prevents duplicate jobs
- **Stuck job recovery:** Jobs stuck in `IN_PROGRESS` for >10 minutes are retried
- **Job types:**
  - `BILL_PARSE` — parse uploaded bill via VPS parser
  - `PRC_PARSE` — parse uploaded PRC document
  - `ZOHO_SYNC_SNAPSHOT` — sync Zoho CRM data
  - `ZOHO_CREATE_CALLBACK_TASK` — create callback task in Zoho
  - `INTAKE_COMPLETED_NOTIFICATION` — email ops when client completes intake
  - `NOTIFICATION_EMAIL` — send queued notification emails
