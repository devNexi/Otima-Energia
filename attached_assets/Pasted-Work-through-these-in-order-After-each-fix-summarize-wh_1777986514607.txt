Bug 1 — Display consumption in kWh, not MWh
The client wants consumption displayed in kWh throughout the portal because it's what employees and clients see on their energy bills and it simplifies payment calculations. The internal storage can stay in MWh, but every user-facing display should convert to kWh (multiply by 1000).
Where to change:

Dossier summary cards ("Consumo Anual" and "Consumo Médio Mensal") — multiply the stored MWh value by 1000 and display with "kWh" label
Dossier detail table rows — same conversion
Dossier edit form — field labels should say "kWh", input values should be in kWh, convert back to MWh (divide by 1000) before saving to the database
RFQ message template — show consumption in kWh
Any other place consumption appears in the UI (deal cards, assembly tab, ECOS)

Example: A dossier storing 18.09 MWh should display as 18.090 kWh. A dossier storing 1.51 MWh monthly should display as 1.510 kWh/mês.
Important: Do NOT change the database storage — keep MWh internally. Only change the display layer and form inputs. Convert on read (×1000 for display) and on write (÷1000 for storage).
Bug 2 — Dossier showing "blocked" even when complete
The client says the dossier shows as blocked even when all fields are filled and the process has been completed. This was partially fixed before but is still happening.
Diagnose: Find every place in the assembly pipeline, blocker engine, and deal progression logic that checks dossier status or completeness. The issue is likely one of:

The dossier status is stuck on "DRAFT" after editing (from the unlock fix we did) and the blocker engine requires "READY" or "LOCKED"
The assembly engine checks a specific field that's null or in an unexpected format
The blocker engine is checking confidence level even though we made it advisory

Search server/blockerEngine.ts, server/deal-assembly-engine.ts, and server/routes.ts for all dossier-related blocker checks. For each one, verify that a fully filled dossier passes the check. If a check requires status "LOCKED" or "READY", change it to also accept "DRAFT" when all required fields are present.
The rule should be: if the dossier has all required fields filled (legal name, CNPJ, distributor, consumption, connection type), it should NOT be a blocker regardless of status or confidence level.
Bug 3 — RFQ template showing "to be confirmed" for filled fields
The RFQ message template was rewritten in Portuguese but is still showing "A confirmar" for distributor, connection type, consumption, and demand — even though these values exist in the dossier.
Root cause: The template function is not receiving the dossier data, or it's reading from the wrong fields. The dossier data might not be fetched when the RFQ dispatch modal opens.
Fix: In the component that generates the RFQ message (BlindAuctionPanel or RFQ dispatch modal), verify that:

The dossier is fetched using the deal's client ID before generating the message
The field names in the template match the actual dossier field names from the API response (check for camelCase vs snake_case mismatches — e.g., averageMonthlyMWh vs average_monthly_mwh)
The dossier object is not null/undefined when the template is built
Add console.log of the dossier object when the modal opens to see exactly what data is available

If the dossier fetch isn't happening, add a query that loads the dossier when the RFQ modal opens. If the fetch is happening but fields are undefined, check the API response shape against what the template expects.

Bug 5 — Can't change operational preferences for suppliers
