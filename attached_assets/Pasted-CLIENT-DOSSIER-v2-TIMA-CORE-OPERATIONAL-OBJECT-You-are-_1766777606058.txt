CLIENT DOSSIER v2 (ÓTIMA — CORE OPERATIONAL OBJECT)

You are working inside an existing production admin portal for Ótima Energia, a Brazilian B2B energy brokerage operating in the Free Contracting Environment (ACL).

This system already includes:

Leads & Clients CRM

Bill upload + OCR extraction

ECOS snapshots

RFO / RFQ tooling

Deal OS (state machine)

Supplier RFQ Adapter (Build #1 in progress)

Your task now is to build Client Dossier v2.

This is a P0 foundational build.
The Client Dossier is NOT just admin data — it is a formal, canonical object used to:

Feed RFQs to suppliers

Standardize supplier interactions

Enforce operational workflow

Signal professionalism and credibility to comercializadoras

No RFQ may be sent without a completed dossier.

🎯 BUSINESS OBJECTIVE

Create a single source of truth for each client’s energy profile that is:

Automatically generated from existing data (OCR, client records)

Manually reviewable and correctable by Ops

Exportable as:

A clean PDF (supplier-facing)

A structured JSON (system-facing)

Hard-gated into the deal workflow

Think of the dossier as:

“The standardized energy load file that Ótima sends to the market.”

2️⃣ DOSSIER GENERATION LOGIC
Automatic Population

When a client has:

At least 1 verified bill upload OR

A completed consumption profile

The system must:

Aggregate last 12 months where available

Calculate:

Annual MWh

Average monthly MWh

Average price (R$/MWh)

Auto-assign eligibility type based on rules

Manual Review & Override

Ops users must be able to:

Edit any calculated field

Add missing values (e.g. demand kW)

Adjust eligibility classification

Add internal notes

All manual overrides must be logged in audit trail.

3️⃣ DOSSIER STATUS & HARD GATES
Status Definitions

DRAFT — incomplete or unvalidated

READY — reviewed and approved for RFQ use

LOCKED — frozen and attached to a Deal / RFQ

Enforcement Rules (CRITICAL)

RFQ cannot be sent unless dossier = READY

When RFQ is sent:

Dossier becomes LOCKED

Snapshot is created (immutable)

Any change requires creating a new dossier version

No silent edits.

4️⃣ DOSSIER SNAPSHOTS (IMMUTABILITY)
New Table: clientDossierSnapshots

Fields:

id

clientDossierId

dealId (nullable)

rfqId (nullable)

snapshotData (full JSON)

createdAt

createdBy

Every RFQ must reference a snapshot, not the live dossier.

This ensures:

Auditability

Supplier dispute protection

Historical accuracy

Constraints:

Portuguese (BR)

Looks professional and standardized

This PDF is attached automatically to RFQs.

7️⃣ WORKFLOW INTEGRATION (MANDATORY)
RFQ Flow Changes

Before sending RFQ:

System checks dossier status

If not READY → block and explain

When RFQ is sent:

Auto-lock dossier

Create snapshot

Attach snapshot to RFQ record
