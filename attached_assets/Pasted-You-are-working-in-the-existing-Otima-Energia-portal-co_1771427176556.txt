B) EXACT STAGE MAPPING RULES (CRITICAL)

Your portal has:

Deal-level stages (state machine): e.g. RFQ_SENT, QUOTES_IN, NEGOTIATION, CONTRACT_SENT, CONTRACT_SIGNED, etc.

Track-level statuses/workflows (each track independent)

We must define deterministic rules so contracts integrate cleanly without breaking the existing deal stage engine.

B1. Principle: Tracks drive Deal Stage (only when appropriate)

A deal can have multiple tracks. The deal stage should reflect the most advanced active track, but must not regress automatically.

Deal Stage Precedence Order (highest wins)

SUPPLY_LIVE
CONTRACT_SIGNED
CONTRACT_SENT
NEGOTIATION
QUOTES_IN
RFQ_SENT
RFQ_READY
DOSSIER_BUILDING
NOVO

B2. When to set Deal Stage based on Track Contract state

We only auto-advance deal stage when:

The track belongs to the deal, is active (not closed/lost)

The resulting stage is ahead of current deal stage

We never auto-regress

Mapping from Track Contract Status → Deal Stage

Track CONTRACT_READY
✅ does NOT automatically change the deal stage
Reason: readiness is internal; deal may still be in negotiation/proposal refinement.

Track CONTRACT_SENT
✅ set deal stage to CONTRACT_SENT IF current deal stage is earlier than CONTRACT_SENT

Track CONTRACT_SIGNED
✅ set deal stage to CONTRACT_SIGNED IF current deal stage is earlier than CONTRACT_SIGNED

Mapping from Track “commercial” progress → Deal Stage (existing)

Do not change existing mapping, but ensure contracts integrate:

If a deal is in NEGOTIATION, and the track contract gets sent, deal advances to CONTRACT_SENT.

If a deal is in CONTRACT_SENT, and signed arrives, deal advances to CONTRACT_SIGNED.

B3. Multi-track rule (very important)

If multiple tracks exist:

Deal stage should reflect the highest-stage active track (using precedence order above).

A track can be CONTRACT_SIGNED while another is still NEGOTIATION. Deal stage should be CONTRACT_SIGNED.

If a signed track is later marked closed/lost (rare), do not auto-regress deal stage. Only manual admin can regress.

B4. Manual overrides

Do not regress deal stage automatically under any circumstance.

All endpoints must be admin-auth protected and follow existing patterns.

D1. Get contract state

D2. Upload supplier contract

Payload:

Server:

D3. Send contract to client (manual now, Clicksign-ready)

Payload:

clientName

clientEmail

clientWhatsapp (optional)

channel: 'email' | 'whatsapp' | 'both'

Rules:

require final supplier contract exists

set contract status to SENT (idempotent; allow resend)

send branded email via nodemailer

return WhatsApp template + wa.me link for UI

audit log CONTRACT_SENT / CONTRACT_RESENT

Deal stage mapping hook:

When contract status becomes SENT:

evaluate deal stage update rules (Section B)

if deal stage is earlier than CONTRACT_SENT → advance to CONTRACT_SENT

if deal stage earlier than CONTRACT_SIGNED → advance to CONTRACT_SIGNED
