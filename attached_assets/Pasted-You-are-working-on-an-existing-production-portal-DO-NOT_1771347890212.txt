# Data Model: Add Deal Tracks (DB + Drizzle)
Create a new table: deal_tracks

Required columns:
- id (uuid / serial, consistent with existing conventions)
- dealId (FK to deals.id, indexed)
- type (enum/string): 'GDL' | 'ACL' | 'ACR' | 'OTHER'
- partnerId (nullable FK to partners.id if partners table exists; for GDL initial partner is Prime/Shell but keep nullable for flexibility)

- id
- trackId (FK deal_tracks.id)
- eventType (string)

- createdAt
- createdByUserId

Uploads/documents: DO NOT add file blobs to DB. Reuse existing upload_sessions / diagnostic_submissions patterns.

# Deal Closure Rules (Important)
- A Deal can have multiple tracks.
- A Deal should be set to CLOSED_WON if any track reaches *_CLOSED_WON (unless business logic says otherwise).
- A Deal should be set to CLOSED_LOST only if ALL tracks are *_CLOSED_LOST (or manually closed).
- Existing Deal stages should not be blocked by tracks; tracks should not force the deal into RFQ stages unless the user explicitly proceeds with RFQ flow.

Implement this carefully:
- Do not auto-mutate the Deal stage aggressively.
- Provide a safe helper function: evaluateDealClosure(dealId) that checks track statuses and updates Deal only when closure conditions are met.

# GDL Eligibility (Data Capture + UI Rules)
For GDL, add an eligibility checklist on the track, based on Prime Energy “Energia por Assinatura” criteria:
We MUST reflect this accurately:

Eligibility required fields/questions:
- averageMonthlyKwh (number; must be >= 500)
- monthsOfHistoryProvided (>= 6) OR proof by uploads
- hasOnsiteGenerationAtLoad (must be false)
- receivesCreditsFromOtherUnits (must be false)
- hasTariffOrTaxExemptions (must be false)

- customerType (PF/PJ)

  - if PJ: CNPJ card
  - LGPD consent evidence or explicit portal consent checkbox (see below)

These fields can be stored in deal_tracks.metadata JSONB for GDL only OR in a dedicated table deal_track_gdl (preferred if you want strong typing).
Choose approach:
- For fastest safe implementation: metadata jsonb + schema validation on backend.
- For stronger long-term: create deal_track_gdl table. If you choose this, do not overcomplicate. Keep minimal.

LGPD consent:
- Add a checkbox “Client consent to share data with Prime Energy for evaluation” with timestamp + user.

# Backend/API Requirements (Express)
Add endpoints:
- POST /api/deals/:dealId/tracks
  body: { type: 'GDL'|'ACL'|'ACR'|'OTHER' }
  returns created track

- GET /api/deals/:dealId/tracks
  returns tracks list

- GET /api/tracks/:trackId
  returns track + metadata + events + linked documents

- PATCH /api/tracks/:trackId
  used to update status, metadata fields, partnerId, notes
  enforce status transitions with a simple per-type transition map (do not overengineer)

- POST /api/tracks/:trackId/events
  for logging events (if not auto-created by PATCH)

- POST /api/tracks/:trackId/submit-to-prime (GDL only)
  This does NOT actually email Prime unless system already does it.
  For MVP: just sets status to GDL_SUBMITTED_TO_PRIME and logs event.
  Gate it with validations:
   - eligibility passed
   - required docs present
   - LGPD consent present

Validation errors must be explicit and helpful.
